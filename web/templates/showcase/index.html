Component identity review checklist

Every UI-affecting PR is reviewed against these five questions. Cite the specific item being addressed or consciously waived in the PR description. Full detail lives in docs/timetrak_ui_style_guide.md (§ Component Identity) and openspec/specs/ui-component-identity/spec.md.

  1. Shape — does the component use the correct shape from the taxonomy? Pills = actions (buttons, timer). Rectangles = status / metadata (chips, badges). Circles = presence dots. (Shape language taxonomy)
  2. Border weight — does every border conform to the two-weight contract? 1px var(--color-border) for structure at rest; 2px var(--color-accent) or var(--color-danger) for state. (Two-weight border contract)
  3. Numerics — does every duration, amount, rate, or count render with font-variant-numeric: tabular-nums? Numeric table columns use .col-num and render right-aligned. (Numeric text contract)
  4. Accent — is the accent color consumed only on an allow-listed surface? (running timer; focus ring; selected-row edge; .btn-primary; links; active nav item; billable / running chips.) Secondary buttons, at-rest non-billable chips, hover states, and at-rest inputs use neutral tokens. (Accent rationing)
  5. State coverage — does every state (default, hover, focused, selected, error, empty, running where applicable) render in /dev/showcase? (Component identity review checklist)

Contribution guide

The showcase is kept honest by two tests in internal/showcase/:

  • Partial coverage — TestComponentCatalogueCoverage enumerates every .html file under web/templates/partials/ and asserts each non-grandfathered stem appears exactly once as a ComponentEntry.PartialName.
  • Snippet integrity — TestSnippetIntegrity asserts every ComponentEntry.PartialName and Example.PartialName resolves against the live template loader.

Adding a new component entry

  1. Add the partial under web/templates/partials/<name>.html following the authoring contract in web/templates/partials/README.md.
  2. Append a ComponentEntry to ComponentEntries in internal/showcase/catalogue.go. Populate PartialName, SourcePath, SpecRef, Purpose, DictKeys, at least one Example, and A11yNotes.
  3. Create a snippet fixture at internal/showcase/snippets/<SnippetID>.tmpl containing the copy-ready {{"{{template"}} "<name>" (dict ...){{"}}"}} call.
  4. Run make test. Coverage + snippet-integrity tests catch any gap.

Adding a new token entry

  1. Propose the token via a foundation change; new tokens are NEVER added in the showcase alone.
  2. Once the token lands in web/static/css/tokens.css, append a TokenEntry to TokenEntries with the right Family (semantic-color | primitive-ramp | spacing | radius | typography | motion | elevation | z-index | breakpoint) and Preview shape.
